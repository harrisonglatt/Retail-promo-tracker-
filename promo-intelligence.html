How baseline + lift are calculated

Plain-English methodology behind every number in this dashboard.

1. Baseline = expected sales if no promo + no display were running

For each SKU, we identify "clean weeks" — weeks where the SKU is genuinely off-promo + off-display:

  • The SKU has sales data (stores > 0)
  • Less than 15% of units sold on promo (filters out always-on Circle redemption noise from Target's feed)
  • No promo event mapped to that SKU in the calendar that week
  • No display / co-space / endcap event that week
  • Not in the SKU's first 2 weeks at 50+ stores (excludes launch ramp distortions)

Calendar events marked "Cancelled," "Rejected," "Not Approved," or "No longer an option" are dropped at parse time. They didn't run, so they don't count as promo weeks + don't pollute baseline classification.

From those clean weeks we compute per-store productivity:

UPS = units / stores ·  SPS = sales / stores

Per-store metrics are critical — a SKU at 200 stores can't be compared to that SKU at 1,700 stores. Raw units would make a launch week look like a flop.

Aggregation method (toggle in sidebar):

MethodWhat it doesWhen to use
Clean (default)Mean of every clean week the SKU hasStable, established SKUs
Trailing 4Mean of the 4 most recent clean weeks before this weekSKUs trending up or down — recent baseline matters more
Trailing 8Mean of the 8 most recent clean weeks before this weekCompromise — recent + smoothed
MedianMedian per-store productivity from all clean weeksNoisy SKUs with a few extreme weeks (outlier-resistant)

Scale back up. For any week we're scoring, we multiply per-store baseline by that week's store count:

baseline units = UPS × stores_this_week
baseline $ = SPS × stores_this_week

That's the volume we'd expect if the SKU performed at its clean per-store rate, given how many stores it was in.

2. Lift = how much actual beat (or missed) baseline

incremental units = actual units − baseline units
incremental $ = actual $ − baseline $
lift % = incremental $ / baseline $

For grouped numbers (a promo type, a SKU summary, a multi-week event), we sum the dollars first, then divide:

group lift % = Σ(incremental $) / Σ(baseline $)

Never average percentages — that gives a wrong answer when weeks have very different baseline sizes.

3. How a week gets bucketed

BucketMeans
BaselineNo promo, no display — the clean reference state
Promo onlyCalendar promo OR ≥15% of units sold on promo, no display
Display onlyCalendar display / co-space / endcap, no promo
Promo + DisplayBoth at the same time — tells you whether the display amplified the price offer
Launch/RampFirst 2 weeks at 50+ stores — held out of promo lift averages so launch noise doesn't drag the number down

4. One caveat worth knowing

Lift % is sensitive to small baselines. A SKU with $20 in clean sales that pops to $60 reads as 200% lift but only $40 incremental — meaningless at the SKU level.

That's why the Data + setup tab has a stability (CV) diagnostic per SKU:

  • CV under 30% = stable baseline, trust the lift number
  • CV 30–60% = moderate noise, interpret with care
  • CV above 60% = noisy baseline, treat lift with skepticism
